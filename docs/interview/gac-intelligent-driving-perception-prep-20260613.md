```text
这个项目的核心是多版本线上交付和风险控制。我们需要在不同版本环境下加载补丁，并且保证执行顺序、版本适配、失败回滚和交付可控。我负责的部分不只是写脚本，还要理解运行环境、版本差异和线上风险。

迁移到智驾场景，我理解算法软件交付也会遇到类似问题，比如不同车型、不同算力平台、不同传感器配置下，模型版本、配置、阈值和依赖库都要可追踪、可回滚、可验证。我的这段经验可以支撑部署调优和交付质量这类工作。
```

### 2. 亚健康自愈系统

讲法：

```text
这个项目主要解决系统进入异常但还没有完全故障的状态时，如何通过指标监控和恢复策略减少人工介入。难点在于怎么定义亚健康、怎么避免误触发、恢复失败怎么办，以及恢复动作之间是否会互相影响。

迁移到感知算法，我理解模型评测和路测问题闭环也类似：需要先定义指标，再发现异常场景，定位是数据、模型、配置、平台性能还是传感器质量问题，最后形成可复现、可回归的闭环。
```

## 技术速补清单

### 必须会讲

- `IoU`：预测框和真值框交并比。
- `Precision/Recall`：误检和漏检的平衡。
- `mAP`：目标检测综合评价指标。
- `FP/FN`：误检、漏检。
- `Latency`：单帧端到端耗时。
- `Throughput`：单位时间处理帧数。
- `Bad case`：模型在特定场景下失败的样本。
- `Calibration`：相机内参、外参、畸变参数、重投影误差。
- `ONNX/TensorRT`：模型部署和推理优化链路。

### 摄像头标定

准备到能讲清：

- 内参：焦距、主点、畸变。
- 外参：相机相对车体或世界坐标系的位置和姿态。
- 畸变：径向畸变、切向畸变。
- 重投影误差：标定质量的重要指标。
- 下线标定：车辆出厂或装配后进行的标定流程。

面试表达：

```text
我目前没有完整量产标定项目经验，但理解标定的目标是把图像坐标、相机坐标、车体坐标或世界坐标之间的关系确定下来。质量上一般会看重投影误差、稳定性和不同环境下的一致性。这个方向我需要结合团队工具链继续补。
```

### 图像质量

准备到能讲清：

- 模糊、过曝、欠曝、眩光、雨雾、脏污、遮挡。
- 分辨率、帧率、动态范围、低照度表现。
- 图像质量问题会影响检测召回、定位精度和稳定性。

### 感知评测

准备到能讲清：

- 分类维度：白天/夜晚、雨雾、隧道、逆光、拥堵、泊车、行人/两轮车/车辆。
- 指标维度：召回率、误检率、漏检率、定位误差、稳定性、延迟。
- 闭环流程：采集 -> 标注 -> 质检 -> 评测 -> bad case 归因 -> 修复 -> 回归。

### 部署调优

准备到能讲清：

- 模型格式：`PyTorch -> ONNX -> TensorRT`。
- 优化手段：算子融合、`FP16`、`INT8`、batch、输入尺寸、前后处理优化。
- 平台指标：CPU/GPU/NPU 占用、显存、内存、延迟、功耗、温度。
- 风险：精度下降、算子不支持、平台差异、量化误差。

### 第二轮：技术初面

**Q1：你怎么理解感知算法评测？**

答：

```text
我理解感知评测不是只看一个总分，而是要按场景、对象类型和问题类型拆开看。比如目标检测可以看 Precision、Recall、mAP、IoU，也要区分误检、漏检、定位偏差和时序抖动。场景上要拆白天、夜晚、雨雾、逆光、隧道、拥堵、泊车等。

工程上我会更关注闭环：数据采集、标注质检、离线评测、bad case 归因、修复后回归。这样才能判断问题到底来自数据、标注、模型、阈值、前后处理，还是部署平台。
```

**Q2：如果车载平台上模型延迟超标，你怎么排查？**

答：

```text
我会先把端到端链路拆开，而不是直接判断是模型慢。一般会分成数据输入、图像预处理、模型推理、后处理、结果发布几个阶段，分别打点看耗时。

然后看资源指标，包括 CPU、GPU/NPU、内存、显存、队列积压、线程调度和功耗温度。如果离线正常、车端异常，就重点看平台差异、输入尺寸、算子支持、量化精度、并发调度和前后处理实现。最后用稳定复现样本和指标证明修复有效。
```

**Q3：你了解相机标定吗？**

答：

```text
我目前没有完整量产标定项目经验，但理解基本目标是确定图像坐标、相机坐标、车体坐标之间的关系。内参主要包括焦距、主点、畸变参数；外参描述相机相对车体或世界坐标系的位置和姿态。质量上会关注重投影误差和不同环境下的一致性。

如果实际做下线标定，我会先熟悉团队已有标定板、采集流程、质检标准和工具链，再补具体实现细节。
```

**Q4：讲一个你做过的问题定位。**

答：

```text
我之前处理过一个 DPDK 相关的 CPU 持续告警问题。它不是单点代码 bug，而是 DPDK 核绑定、恢复轮询逻辑和系统资源调度之间发生冲突。

我当时先看监控指标确认告警是否真实，再缩小到具体进程和线程，看 CPU 绑核和轮询行为，最后定位到恢复逻辑和 DPDK 运行模型冲突。这个经验和车载平台调优比较像：要先把指标拆清楚，再定位是资源、调度、链路还是业务逻辑问题。
```

**Q5：如果模型在夜间漏检行人，你怎么闭环？**

答：

```text
我会先确认这是个别样本还是系统性问题。第一步按场景收集样本，比如夜间、逆光、雨雾、低照度、遮挡。第二步检查标注质量，确认真值没有问题。第三步看模型输出，包括置信度、框位置、类别混淆和前后处理阈值。第四步评估图像质量，比如曝光、模糊、噪声和摄像头状态。

归因之后再决定动作：如果是数据覆盖不足，就补采和重训；如果是阈值或后处理问题，就调参回归；如果是图像质量问题，就和摄像头或标定链路一起排查。最后必须用固定测试集和新增 bad case 做回归。
```
