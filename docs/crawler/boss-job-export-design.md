# Boss 职位采集导出设计

## 目标

在 Boss 职位列表页提供只读职位采集能力，按当前页面搜索条件自动翻页采集岗位信息，并导出 CSV。采集结果包含所有岗位及现有非 AI 筛选链路的通过、过滤或错误状态，方便用户在表格工具中继续筛选。

## 当前缺口

- 现有批量流程会读取职位列表、职位详情并执行投递，但没有只读导出模式。
- 当前职位信息只展示在插件 UI 和投递日志中，无法直接导出为表格。
- 现有筛选链路和投递动作耦合，不能直接用于“只判断、不投递”的采集场景。

## 已锁定决策

- 采集器只使用当前已登录 Boss 页面、当前搜索条件和现有分页能力，不实现全站爬取。
- 默认最多采集 50 页，每采集 1 个岗位后等待 3 秒，每页翻页后等待 5 秒。
- 采集会持续到页数上限、列表为空、列表无变化或用户手动停止，不尝试绕过平台限制。
- 采集时按 `encryptJobId` 去重。
- 输出格式为 CSV 下载，文件名使用 `boss-jobs-YYYYMMDD-HHmmss.csv`。
- 采集流程完全禁止调用投递、加好友或打招呼接口。
- CSV 导出常用字段：职位名、公司、薪资、城市、区域、商圈、详细地址、经验、学历、职位标签、福利、岗位描述、HR 名称、HR 职位、活跃时间、职位链接、筛选状态、筛选原因、采集错误。
- 采集导出不调用 AI 过滤，即使用户已开启 AI 过滤也只执行非 AI 的只读筛选判断。
- 采集按钮与投递按钮互斥，避免两个流程同时操作页面分页和详情读取。

## 非目标

- 不绕过登录、验证码、风控或平台访问限制。
- 不做独立后端爬虫、全站爬取或未授权接口扫描。
- 不新增 JSON 导出、插件内历史列表或 UI 可配置页数/间隔。
- 不在采集导出中执行 AI 过滤或生成 AI 分析结果。
- 不改变现有批量投递行为和投递上限逻辑。

## 数据或状态变化

- 运行时新增采集状态：是否采集中、是否请求停止、当前页数、已采集数量、失败数量、状态文本。
- 采集结果仅保存在内存中，完成后下载 CSV；不写入浏览器 storage。
- 不写入已投递公司、已投递 HR、投递统计或投递日志。

## API 或页面影响

- 职位页统计控制区新增“采集导出”和“停止采集”按钮。
- 采集时展示当前页、已采集数量、失败数量和状态。
- 采集完成后自动触发 CSV 下载。
- 无后端 API 变化。

## 实施顺序

1. 新增只读采集 composable/store，复用 `jobList` 和 `usePager`。
2. 新增只读筛选判断入口，复用现有非 AI before handlers，但不执行投递请求、AI 过滤和 after handlers。
3. 新增 CSV 字段映射、转义和下载逻辑。
4. 在职位页控制区接入采集按钮、停止按钮和进度文案。
5. 运行类型检查和 lint。

## 验收标准

- 点击“采集导出”后，最多采集当前搜索条件下 50 页职位并下载 CSV。
- CSV 可被表格软件打开，中文、逗号、双引号和换行不破坏列结构。
- CSV 包含所有采集到的岗位，并带筛选状态和筛选原因。
- 即使开启 AI 过滤，采集导出也不会调用 AI 模型，CSV 不包含 AI 过滤分数、响应或 Prompt。
- 采集期间不会触发自动打招呼、投递、加好友或写入已投递缓存。
- 采集期间每处理 1 个岗位后至少等待 3 秒，避免在单页内连续快速读取详情。
- 采集期间投递按钮不可用；投递期间采集按钮不可用。
- 用户点击“停止采集”后停止翻页，并导出已采集数据或提示无数据。
- 列表为空、列表无变化或详情读取失败时，流程能结束并给出状态。
